PJF runs a self-hosted homelab — two main nodes (dsfitz storage + t480 compute) plus n54l as the off-site backup target. Read this first when asked about the selfhosting setup/config, then the per-service files. **Verify live state, don't assume** — `docker ps` on t480 and `ssh dsfitz 'sudo /usr/local/bin/docker ps'` on the NAS (sudo is passwordless for docker only).

- **dsfitz** — Synology **DS1821+** (Ryzen V1500B 8-thread), kernel 4.4, reached via Tailscale + LAN **192.168.8.184**. The **storage/RAID + backup** node. **RAM-constrained: only 3.8 GB**, runs ~15 containers (Plex, Immich + ML, qBittorrent, Calibre-Web-Automated, lubelogger, …) and chronically swap-thrashes. **The *arr apps (Prowlarr/Sonarr/Radarr/Lidarr) moved to t480 on 2026-06-23** — only **qBittorrent** (the download client) and **Plex** remain here from that stack; bulk media + downloads still live here under `/volume2/data`. **FreshRSS migrated to t480 on 2026-06-27** (old container left stopped, restart=no); see [[project_freshrss]]. **Don't add heavy stacks here.** Docker via `/usr/local/bin/docker` (Container Manager). Data under `/volume2/docker/<service>` and `/volume2/shared-folder1`.
- **t480** — ThinkPad, **always-on server** (treat it as a server, not a laptop), 8 cores / **22 GB RAM**, Ubuntu. The **compute** node for RAM/CPU-hungry services. LAN **192.168.8.228**, tailnet `t480.astrapia-degree.ts.net`. **It is NOT vulnerable to laptop problems — do not raise sleep / suspend / lid-close / battery / WiFi-dropout as reliability concerns.** It's configured to stay on permanently (lid-close is a no-op, suspend disabled). PJF has corrected this repeatedly; never suggest "disable suspend" or treat it as a flaky laptop again.
- **n54l** — HP ProLiant N54L, **Ubuntu 24.04**, at **Black Rock** (off-site, different LAN `192.168.0.11`), tailnet IP `100.100.21.95`, user `pjf`. The **off-site backup target** (long-planned, stood up 2026-06-24). 7.6 GB RAM (idle), `/backups` = RAID1 2×3 TB (spare 3 TB `sdd` + 1 TB `sda` for later expansion). Has `restic`/`sshfs`/`tc`; no `trickle` (dropped from 24.04). Runs the **Immich off-site backup** (`~/ops/immich-backup.sh`, cron 03:30): PULL model — restic on n54l pulls dsfitz's Immich UPLOAD_LOCATION over read-only sshfs (Tailscale, direct ~26 ms) into a local encrypted repo `/backups/immich/repo`. Keeps heavy work off the RAM-starved dsfitz. **Gotcha:** dsfitz's SFTP namespace is chrooted, so over sshfs the path is `dsfitz:/docker/immich/library` (= `/volume2/docker/immich/library` in the shell) — see [[dsfitz-scp-sftp-path]]. Full write-up: `~/pkm/zettelkasten/immich n54l backup.md`.

**Architecture pattern** (used by Nextcloud, Paperless, Immich, the *arr stack, Audiobookshelf): run the app + its DB on **t480's local SSD**, keep bulk data on **dsfitz over NFS** via Docker `local`-driver NFS volumes (`type: nfs`, `addr=192.168.8.184`, `nfsvers=3`) so the container fails fast if the NAS is down. SQLite/DB stays local (never on NFS). Compose stacks live in `~/docker/<service>/` (secrets in `.env`, mode 600). NFS exports configured in DSM per shared folder, scoped to t480's IP. **Gotcha:** an export can be `ro` and/or `all_squash` in DSM (reads succeed but writes fail with EROFS / blocked dirs) — for any app on t480 that must *write* to NAS storage, the DSM NFS rule for 192.168.8.228 needs **Read/Write + Squash "No mapping"** (the `/volume2/data` rule was fixed this way for the *arr migration so the container uid 1026 passes through as patrick_admin).

**Off-site backups to n54l (built 2026-06-27):** three nightly restic-pull jobs on n54l, all encrypted local repos on `/backups`. (1) **Immich** photos `03:30` → `/backups/immich/repo`. (2) **services** `04:30` → `/backups/services/repo`: pulls self-hosted app data/config over sshfs from BOTH t480 (`~/docker/*`) and dsfitz (`/docker/*`, `/data/books`) — karakeep, freshrss, memos, *arr, audiobookshelf, nextcloud app/config, paperless, lubelogger, calibre+books. (3) **restic-t480 copy** `05:00` → `/backups/restic-t480`: off-site mirror of the dev-app DB repo that t480-ops pushes to dsfitz. Excluded: 14 TB media, podcasts, torrents, nextcloud user files (backed up via other means — recovery path still undocumented). Scripts/env in n54l `~/ops/`; new repo password (services+restic-t480) must be in PJF's password manager. **Live progress check:** `ssh n54l '~/ops/backup-status.sh'` — reports per-job pid/elapsed and read/write throughput by sampling restic's own `/proc/<pid>/io` (no sudo/repo password); prints "not running" when idle and warns on pile-ups. Note restic streams data over sshfs, so progress shows in `rchar`/`wchar`, not block-device `read_bytes`. Full write-up [[services n54l backup]]; decisions/restore doc `~/pkm/homelab-backup.html`; tick-list `~/pkm/homelab backup checklist — what to send to n54l.md`.
